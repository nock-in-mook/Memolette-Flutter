理屈上はそう思えるんだけど、**SQLite + Google Drive = 定番の落とし穴**で、高確率でDB壊れる。

**壊れる理由**
- claude-mem は SQLite DB (`~/.claude-mem/claude-mem.db`) + FTS5 + 必要に応じて ChromaDB を使う。
- Google Drive / Dropbox は「ファイルが閉じたタイミング」でアップするイメージだけど、SQLite は Worker Service が常時 DB を開きっぱなしにする設計。
- ドライブ側は中途半端な状態のファイルを同期しちゃう → Journal ファイルとのずれで DB 破損。
- さらに 2 台で同時〜短時間差で編集すると、Google Drive が `claude-mem.db (1)` みたいな衝突ファイルを作って、どちらが正か分からなくなる。

これは SQLite 一般の話で、公式も「ネットワーク/同期ドライブに SQLite を置くな」と明言してる。`~/.claude-mem/` を Google Drive 配下にシンボリックリンクするとほぼ確実に事故る。

**現実的な選択肢**

| 方針 | 難易度 | 安全性 |
|---|---|---|
| A. **PC ごとに別メモリとして使う**（同期しない） | 楽 | ◎ |
| B. **セッション終了時に export → Drive → 別PCで import** | 中（スクリプト化要） | ○ |
| C. **git リポジトリに DB を入れて push/pull** | 中 | △ DB肥大化の罠 |
| D. **claude-mem が将来出すクラウド同期機能を待つ** | — | — |

個人的には **A が無難**。のっくりさんの場合、Mac と Windows で作業文脈がそもそも別（Mac=Flutter開発、Windows=その他も多い）だから、メモリも別の方が混線しないかも。
