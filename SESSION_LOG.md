### Firestore（メモ・タグ・ToDo 同期）

**1ユーザー想定（メモ500・タグ30・ToDoリスト10、端末2台、1日アプリ起動5回・編集75回）**:
- writes: ~75/日 = 月 2,250
- reads: ~5,500/日 = 月 16万（起動時の初回 snapshot が支配的）
- ストレージ: ~0.5MB

**Blaze プラン換算**: 1ユーザー **約 $0.10/月**（10〜15円）

**Spark（無料枠）**: 1ユーザーなら全部余裕、1万人で reads/writes が枠超え

### Firebase Storage（画像同期で追加）

**1ユーザー画像100枚・平均150KB = 15MB**:
- ストレージ: 月 $0.0004
- 帯域（端末間DL分）: 月 $0.005

**1000ユーザー合計でも 月 $6**。誤差レベル。

### 削減レバー

起動時の全件 snapshot が reads を支配してる。**Firestore のオフラインキャッシュ**（FlutterFire で大体デフォルト ON）が効いてれば、変更分だけ reads 消費に変わる → reads/日が 5500 → 100〜200 まで落ちる可能性。今 ON か後で確認しとく。

### 結論

Pro サブスク想定価格（月数百円）に対して原価は1ユーザー月15円程度。**全然余裕**。1万人規模でも Blaze で月 $1,500、サブスク 5% でも回収余裕。

### 画像同期の設計案

**方針**: Firebase Storage に画像本体、メモ doc に `imageUrls: [...]` を含める

**Storage パス**: `users/{uid}/memos/{memoId}/{imageId}.jpg`

**フロー**:
- 画像追加 → ローカル MemoImages 追加 → アップロード時に Storage へ送信して URL 取得 → メモ doc の `imageUrls` に追加
- ダウンロード時は doc の URL リストを見て、未取得分だけ Storage から DL → ローカル MemoImages 追加

**スコープ判断**:
- A. **画像本体 + メモ doc の URL リスト同期**（メモ編集時に新規画像があれば Storage へ）
- B. A + **画像個別の削除同期**（メモ単位ではなく画像単位）

A から始めるのがいい。B は削除同期タスクと一緒で OK。

「無料ユーザーが大量にアップロードして帯域・容量を食い潰す」のが懸念。**標準的な対策は容量上限制**：

```
無料版:    合計 100MB（約500-700枚）
Pro 月額500円:  1GB（約7000枚）
Premium 月額1500円: 10GB（約7万枚）
```

これなら：
- 無料ユーザーは数年使っても上限内
- ヘビーユーザーは Pro / Premium へ誘導 → 原価カバー
- ユーザーごとに「あと○MB」表示でわかりやすく

容量カウントは Firestore に metadata として記録すれば実装可能。Phase 4（サブスク実装）と一緒にやるのが自然。今は同期機能だけ作って、上限実装は後回しで OK。
